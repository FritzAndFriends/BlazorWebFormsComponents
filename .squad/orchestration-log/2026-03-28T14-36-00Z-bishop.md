## Implementation Details

### SkinFileParser Architecture

**Parsing Pipeline:**
1. Strip ASP.NET comments: `<%--.*?--%>` via regex
2. Wrap content in `<root>...</root>` for valid XML
3. Replace prefixes: `<asp:` → `<asp_` for XML-safe element names
4. Parse with XDocument and walk tree

**Public API:**
- `ParseSkinFile(string skinContent, ThemeConfiguration config = null)`
- `ParseSkinFileFromPath(string filePath, ThemeConfiguration config = null)`
- `ParseThemeFolder(string folderPath, ThemeConfiguration config = null)`

**Type Mappings:**
- `BackColor`, `ForeColor`, `BorderColor` → `WebColor.FromHtml(value)`
- `BorderWidth`, `Height`, `Width` → `new Unit(value)`
- `Font-Size` → `FontUnit.Parse(value)`
- `Font-Bold`, `Font-Italic` → `FontInfo` boolean properties
- `BorderStyle`, `HorizontalAlign`, `VerticalAlign` → `Enum.TryParse<T>()`

**Sub-Styles:**
- Nested elements (e.g., `<HeaderStyle>`) become entries in `ControlSkin.SubStyles` dictionary
- Case-insensitive matching (e.g., "HeaderStyle" == "headerstyle")
- Returns TableItemStyle with style properties + alignment

### Design Decisions

1. **XML parsing with preprocessing** — Leverages .NET's robust XDocument vs custom lexer
2. **No external dependencies** — Uses only System.Xml
3. **Defensive error handling** — Allows partial parse success; silently ignores unknown attributes
4. **Nested sub-styles naturally handled** — XML tree walk handles arbitrary nesting
5. **Case-insensitive matching** — Matches Web Forms robustness

### Alternatives Considered & Rejected

1. **Custom lexer/parser** — More complex, higher maintenance burden
2. **Regex-based extraction** — Fragile with nested elements and complex attributes
3. **HTML Agility Pack** — Additional dependency, overkill for this use case

# Recipe: Missing Code-Behind File

## Error Signature

```
CS0103: The name 'ControlName' does not exist in the current context
```

Multiple `CS0103` errors in a single `.razor` file, all referencing control IDs or event handlers that would normally be in a code-behind.

## Detection

If a `.razor` file has `@ref="X"` attributes but no matching `.razor.cs` file exists, the code-behind was suppressed.

```powershell
# Find .razor files with @ref that have no .razor.cs
Get-ChildItem -Recurse -Filter *.razor | Where-Object {
    (Select-String -Path $_.FullName -Pattern '@ref=' -Quiet) -and
    -not (Test-Path "$($_.FullName).cs")
}
```

## Root Cause

The CLI's `PageCodeBehindEmissionPlanner` may demote a code-behind to `migration-artifacts/codebehind/PageName.razor.cs.txt` when it detects patterns it considers unsafe for the compile surface. Meanwhile, `ComponentRefMarkupTransform` already added `@ref` attributes to the markup (it runs before the emission planner decides).

**Result:** Markup has `@ref="CartList"` pointing to fields that don't exist because the `.razor.cs` was never emitted.

## Fix Pattern

1. **Check the artifact folder** for the suppressed code-behind:
   ```
   migration-artifacts/codebehind/PageName.razor.cs.txt
   ```

2. **Copy the artifact to the compile surface:**
   ```powershell
   Copy-Item migration-artifacts/codebehind/PageName.razor.cs.txt PageName.razor.cs
   ```

3. **Fix compile errors** in the recovered code-behind — common issues:
   - `new DbContext()` → see `new-dbcontext-to-di.md` recipe
   - `InnerText` on HTML elements → see `innertext-to-markup.md` recipe
   - Missing logic classes → create stubs or inject services

4. **If no artifact exists**, create a stub code-behind by scanning the `.razor` for references:
   - Every `@ref="X"` needs a `private ComponentType X = default!;` field
   - Every `OnClick="Handler"` needs a `private void Handler() { }` method
   - Every `@SomeProperty` binding needs a corresponding property

## Stub Template

```csharp
using Microsoft.AspNetCore.Components;
using BlazorWebFormsComponents;

namespace MyApp;

public partial class PageName : WebFormsPageBase
{
    // Fields for @ref bindings (generated by ComponentRefMarkupTransform)
    private GridView<ItemType> GridName = default!;
    private Button BtnName = default!;
    private Label LblName = default!;

    // Event handlers referenced in markup
    private void BtnName_Click() { /* TODO: implement */ }

    // Data loading
    protected override async Task OnInitializedAsync()
    {
        await base.OnInitializedAsync();
        // TODO: load data
    }
}
```

## Prevention

This issue occurs less frequently as the emission planner is updated to recognize BWFC-supported patterns (e.g., `GridViewRow`, `FindControl`, `InnerText`) as safe. If you see this pattern, check whether the planner's unsafe-pattern list is overly conservative for patterns that BWFC shims support.

## Verification

After creating or recovering the `.razor.cs`, all `CS0103` errors for that page should resolve. Run `dotnet build` to confirm.
